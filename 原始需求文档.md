# 乐信产品AI工作平台 PRD

## 1. 产品概述

### 1.1 产品名称
乐信产品AI工作平台

### 1.2 产品定位
面向产品经理的AI辅助工作平台，通过AI能力帮助产品经理高效完成需求文档撰写和UI设计生成工作。

### 1.3 目标用户
- 乐信集团产品经理
- 产品设计师
- 业务分析师

### 1.4 核心价值
- 提升PRD文档撰写效率
- 降低UI设计沟通成本
- 标准化产品输出物
- 游戏化激励机制提升使用积极性

---


## 3. PRD需求模块

### 3.1 PRD工作台（新建需求）

#### 3.1.1 工作流程
采用四步骤流程设计：

| 步骤 | 名称 | 功能描述 |
|------|------|----------|
| Step 1 | 需求输入 | 用户输入原始需求描述，AI提供智能建议 |
| Step 2 | PRD文档 | AI生成PRD文档，支持实时编辑 |
| Step 3 | 智能审查 | AI对PRD进行质量评分和改进建议 |
| Step 4 | 应用导出 | 导出到TAPD或关联到UI设计 |

#### 3.1.2 Step 1 - 需求输入
**功能点：**
- 大面积文本输入区域（自适应屏幕高度）
- AI模型选择（Claude/GPT等）
- 可折叠的AI智能建议面板
- 建议类型：功能完善、边界场景、用户体验、技术可行性

**交互说明：**
- 输入框占据主要空间，最小高度300px
- 点击"收起建议"可折叠右侧建议面板
- 点击建议项可快速插入到输入框

#### 3.1.3 Step 2 - PRD文档生成

**功能点：**
- AI自动生成结构化PRD文档
- 实时编辑能力
- 基于预设约束生成高质量文档

**AI生成约束与设定：**

##### 角色设定（Role）
AI扮演一位拥有10年经验的资深产品经理和全栈系统架构师，擅长将模糊的需求转化为逻辑严密、可落地的PRD需求文档。

##### 任务定义（Task）
基于用户提供的简短需求描述，通过深度推理，扩展成一份完整的《产品需求文档 (PRD)》。

##### 输出结构要求（Output Requirements）
生成的PRD文档必须按以下结构输出 Markdown 格式：

**1. 产品概览-项目背景与目标**
| 字段 | 说明 |
|------|------|
| 产品/需求名称 | 对需求进行总结命名 |
| 需求价值 | 一句打动用户的核心价值主张 |
| 核心痛点 | 一针见血地指出要解决的问题 |

**2. 用户画像**
- 描述 1-2 个典型用户角色（Persona）
- 包含用户痛点和使用场景

**3. 功能列表/业务流程逻辑**
- 使用 MoSCoW 原则分类：
  - Must have（必须有）
  - Should have（应该有）
  - Could have（可以有）
  - Won't have（暂不做）
- 每个功能必须包含"页面/模块"归属

**4. 接口需求详情**
- 描述每个功能的具体接口需求
- 包含：HTTP 方法、URL 路径、请求参数、响应格式
- 示例：`POST /api/items` (创建)，`GET /api/items` (查询)

**5. 页面与交互架构**
- 列出核心页面结构（首页、详情页、设置页等）
- 描述关键页面的核心组件

**6. 数据模型与埋点建议**
- 以技术视角列出核心数据库实体（Entities）
- 列出关键字段（Fields）
- 示例：`User (ID, Name, Avatar)`, `Item (ID, Name, Status)`

**7. 视觉风格指导**
- 设计原则：遵循「轻松愉悦」、「简单易用」、「专业信赖」三大设计价值观
- 结合可用性基准（Fitts定律、格式塔原则）
- 确保功能逻辑与用户心智模型匹配

##### 视觉规范约束

**主色调分布：**
| 颜色 | 色值 | 用途 |
|------|------|------|
| 品牌蓝 | #4D83FF | 主要操作按钮、重要链接、标题 |
| 洋红 | #FF4D82 | 警告、错误状态 |
| 辅助红 | #E53948 | 删除操作、强调重要信息 |
| 橙色 | #FF9D33 | 警告、提示用户注意 |
| 绿色 | #26BF71 | 成功状态、确认操作 |
| 灰色 | #1E2533 | 背景、文本、边框 |

**字体规范：**
| 平台 | 中文字体 | 英文字体 |
|------|----------|----------|
| iOS | 苹方 (PingFang SC) | Roboto |
| Android | 思源黑体 (Noto Sans CJK SC) | Roboto |
| Windows | 微软雅黑 (Microsoft YaHei) | Segoe UI |

> 设计稿以iOS平台为准，使用苹方 (PingFang SC) 及 Roboto

##### 语言风格约束
- 使用简单、直接的表达方式
- 避免复杂或生僻的词汇
- 句子之间逻辑关系清晰
- 避免复杂的语法结构
- 将过渡词和连接词替换为最基础、最常用的词语

#### 3.1.4 Step 3 - 智能审查
**功能点：**
- PRD质量评分（0-100分）
- 多维度评估：
  - 完整性
  - 清晰度
  - 可行性
  - 一致性
- 具体改进建议列表
- 一键优化功能

#### 3.1.5 Step 4 - 应用导出
**功能点：**
- 同步到TAPD（一键创建TAPD需求）
- 关联到UI设计（跳转UI模块并带入PRD内容）
- 标记为已完成
- 导出为Markdown文件

### 3.2 我的需求（需求列表）

#### 3.2.1 列表功能
- 卡片式布局展示
- 筛选：全部 / 进行中 / 已完成
- 搜索：按标题和描述搜索
- 统计：显示需求总数

#### 3.2.2 卡片操作
| 操作 | 图标 | 功能 |
|------|------|------|
| 编辑 | Edit2 | 展开编辑区域，修改PRD内容 |
| 查看 | Eye | 展开查看PRD详情 |

## 5. 全局功能

### 5.1 用户信息展示

**Header区域显示：**
- 用户头像和姓名
- OA账户关联状态
- 本周使用时长
- Token消耗量
- 排行榜名次

**Sidebar区域显示：**
- 用户头像
- 用户姓名
- OA账户关联状态

### 5.2 Toast通知系统
**类型：**
- success（绿色）：操作成功
- error（红色）：操作失败
- warning（黄色）：警告提示
- info（蓝色）：信息提示

**交互：**
- 右上角滑入显示
- 3秒后自动消失
- 支持手动关闭

### 5.3 状态管理
**PRD状态：**
- in_progress（进行中）
- completed（已完成）

**UI设计状态：**
- in_progress（进行中）
- completed（已完成）

- 背景白色

- 圆角8px
- hover阴影效果

**输入框：**
- 边框1px #EDEDEE
- 圆角8px
- focus边框变主色

## 9. 交互流程

### 9.1 PRD创建流程
```
用户输入需求 → AI生成PRD → 智能审查 → 导出/关联UI
     ↓              ↓           ↓
  AI建议辅助    实时编辑    质量评分
```

### 9.2 PRD到UI关联流程
```
PRD完成 → 点击"开始UI设计" → 跳转UI模块 → 自动填入PRD内容 → 生成UI设计
```

### 9.3 UI设计调整流程
```
初始设计生成 → 对话式调整 → AI重新生成 → 满意后导出
```
